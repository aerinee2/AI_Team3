# 🚨 SafeEat Korea: AI Food Label Scanner for Travelers

> **현실의 수많은 오인식 한계들을 하이브리드 파이프라인(EasyOCR + Donut)과 스마트 후처리 사전으로 치열하게 풀어낸 외국인 여행자 대상 식품 성분 분석**

---

## 🛠️ Tech Stack
- **Frontend:** HTML5, CSS3, JavaScript
- **Backend:** FastAPI (Python)
- **AI Core:** EasyOCR & Donut VLM (`naver-clova-ix/donut-base` 기반 자체 데이터 학습 어댑터)
- **Infrastructure:** 구글 코랩 T4 GPU + ngrok 포워딩

---

## 📌 3.3 FastAPI 시연 어플리케이션 제작 & 시행착오의 기록



### 1️⃣ 인프라 변천사: 8시간의 벽과 GPU 비용과의 싸움
- **Step 1. ngrok의 한계 발견:** 처음에는 FastAPI에 화면만 얹어서 `ngrok`으로 띄웠으나, 무료 플랜 특성상 **8시간마다 주소가 계속 바뀌는 문제**가 있었다. 시연이나 팀원 간 공유를 하기엔 서비스의 영속성이 너무 떨어지는 한계가 존재했다.
- **Step 2. Hugging Face Spaces로 이사:** 로컬 컴퓨터가 꺼져도 24시간 상시 돌아가는 웹페이지를 구축하고자 `Hugging Face Spaces` 인프라로 전면 이관하였다. 이 단계에서 경량화 엔진인 EasyOCR을 붙여 기본 서빙 파이프라인의 기초 작동 메커니즘을 성공적으로 검증하였다.
- **Step 3. GPU 비용 문제와 최종 하이브리드 정착:** 그러나 본격적으로 학습된 **Donut 모델**을 얹으려니 필수적으로 GPU 자원이 요구되었다. 허깅페이스 스페이스에서 GPU 인프라를 업그레이드하려 했으나 **최소 15달러부터 강제 크레딧 충전**해야 하는 부담이 있었다. 결국 이를 우회하기 위해, **구글 코랩의 T4 GPU 연산 자원과 ngrok의 고정 도메인을 결합**하여 자원 비용 0원으로 대용량 모델을 안정적으로 돌릴 수 있는 하이브리드 서버 환경을 최종 구축하였다.

### 2️⃣ 데이터 인식의 한계와 Hybrid AI의 도입
- **EasyOCR 단독 테스트 단계:** 초기 단계에서는 EasyOCR만 넣어서 시험 삼아 돌려보았다. `'우유함유'`, `'대두함유'` 같이 글씨가 크고 정형화된 명시적 단어들은 잘 잡아냈으나, 식품 패키지가 조금만 복잡해지면 `'고추장함유'`, `'해바라기씨유'` 같이 텍스트 밀도가 높거나 폰트가 왜곡된 미세 성분 글자들은 제대로 인식하지 못하고 누락되는 한계를 보였다.
- **Donut 멀티모달 결합 단계:** 이를 보완하기 위해 식품 라벨 데이터로 학습시킨 **Donut 모델**을 2차 교차 검증 엔진으로 결합하였다. Donut은 글자 선만 보는 것이 아니라 이미지 전체의 문맥과 레이아웃을 통으로 이해하기 때문에, EasyOCR이 시각적으로 완전히 놓친 미세 성분이나 조미료류까지 한 번 더 짚어내 주며 원재료명 인식 흐름의 최종 정확도를 한 단계 더 끌어올릴 수 있었다.

### 3️⃣ 후처리 버그 (Troubleshooting)
인공지능 모델이 텍스트를 읽어온 뒤에도 현실 데이터에는 수많은 오역과 필터링 버그가 존재하여 이를 코드로 직접 방어해야 했다.
- **알레르기 항원 전면 통합:** 초코파이 같은 가공식품을 돌려보면 성분표에 외국인들에게는 매우 낯선 한자어인 `돈지(돼지기름)`, `돈피(돼지껍질/젤라틴)` 같은 단어들이 찍힌다. 환자의 직관성과 앱의 기획 의도에 맞추기 위해 백엔드 사전을 수정하여 **`돈지/돈피/돈육 ➡️ 돼지고기`**, **`우육/우지/비프 ➡️ 쇠고기`**로 묶어서 하나의 대표 알레르기 유발 항원으로 정제되도록 로직을 유도했다.

## 📊 과정에서 배운 점 (Retrospective)

둥글고 반사율이 높은 비닐 포장지 특성상 실제 EasyOCR 로그를 뜯어보면 `춤0600H| 양 0 쭉...` 같은 외계어 노이즈가 폭주하곤 한다. 

현실 세계에서 완벽한 100%의 인식률을 가진 인공지능 모델은 없기에, 본 프로젝트는 **"AI 모델이 터트린 에러나 오타를 뒤쪽 파이썬 후처리 로직(`CORRECTION_MAP`)과 프론트엔드의 토글 격리 UI로 어떻게 방어하고 정제해 낼 것인가"**에 깊게 집중하였다. 

단순히 완성된 모델을 가져다 쓰는 것을 넘어, 현실적인 오인식 한계들을 하이브리드 파이프라인과 엔지니어링 예외 처리로 하나씩 메워가는 값진 트러블슈팅 과정을 경험하였다.
