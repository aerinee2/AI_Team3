SafeEat Korea: AI Food Label Scanner for Travelers
Donut VLM 단독 엔진(Free OCR)과 후처리 로직을 활용한 외국인 여행자 대상 식품 성분 분석 시스템

Tech Stack
Frontend: HTML5, CSS3, JavaScript

Backend: FastAPI (Python)

AI Core: Donut VLM (naver-clova-ix/donut-base 기반 파인튜닝 어댑터) [Free OCR]

Infrastructure: 구글 코랩 T4 GPU, ngrok 포워딩

3.3 FastAPI 시연 어플리케이션 제작 & 시행착오의 기록
1️⃣ 인프라 변천사 및 구동 환경 최적화
Step 1. ngrok 도메인 변동 한계: 초기에는 FastAPI에 프론트엔드를 얹어 ngrok으로 배포했으나, 무료 플랜 특성상 일정 시간마다 접속 주소가 변경되어 서비스 영속성이 떨어지는 문제가 있었다.

Step 2. Hugging Face Spaces 이관 테스트: 상시 구동 환경을 구축하고자 Hugging Face Spaces 인프라로 전면 이관하여 기본 파이프라인의 정상 작동을 검증하였다.

Step 3. T4 GPU 및 고정 도메인 결합: 이후 Donut 모델을 구동하기 위해 GPU 자원이 필수적으로 요구되었으나, 허깅페이스 환경의 GPU 업그레이드는 비용 부담이 존재했다. 이를 우회하기 위해 구글 코랩의 T4 GPU 자원과 ngrok의 고정 도메인을 연동하여 추가 비용 없이 VLM 모델을 서빙할 수 있는 서버 환경을 최종 구축하였다.

2️⃣ 데이터 인식의 한계와 Free OCR (Donut 단독 엔진) 도입
하이브리드(EasyOCR) 단계의 한계: 초기 설계에서는 글자 선을 검출하는 OCR과 문맥을 파악하는 Donut을 혼합하여 사용했다. 그러나 식품 패키지 표면의 조명 반사로 인해 명암비를 높이자, OCR 엔진이 비닐의 구김을 텍스트 노이즈(춤0600H| 등)로 잘못 인식하여 결과 화면의 가시성을 저해하는 한계가 발생했다.

Donut 단독 엔진(Free OCR) 전환: 이를 해결하기 위해 전통적인 OCR 단계를 제거하고, 레이아웃과 문맥을 통합적으로 이해하는 Donut VLM 단독 추론 체제로 전환하였다. 

3️⃣ 후처리 로직 및 데이터 동기화 (Troubleshooting)
모델 추론 이후 발생하는 데이터 표기 변형과 프론트-백엔드 간 통신 문제를 해결하기 위해 예외 처리 코드를 반영했다.

알레르기 항원 매칭 사전 통합: 원재료명에 포함된 돈지(돼지기름), 전신(돼지), 우육 등 파편화된 표현들을 직관적인 알레르기 경고로 변환하기 위해, 백엔드 정제 사전을 거쳐 돈지/돈피/돈육 -> 돼지고기, 우육/우지/비프 -> 쇠고기와 같이 대표 항원명으로 표준화되도록 로직을 구성했다.


추후 보완하면 좋을 내용
다국어 확장: 영어권 외에 다양한 언어(중국어, 일본어, 아랍어 등) 대응을 위한 다중 번역 모듈 추가.

문자열 정제 고도화: 전처리 단계에서 한국어 결합 구조를 벗어난 깨진 문자열 패턴을 자동 필터링하는 알고리즘 도입.
